🌍 ARQUITECTURA DE DIRECTORIOS - GOLAND PLATFORM MVP (SUPABASE / POSTGRESQL ONLY)

Este documento describe la estructura y distribución física de las carpetas y
archivos del proyecto GOland, detallando las responsabilidades de cada capa.
Nuestra persistencia en producción y desarrollo utiliza única y exclusivamente
Supabase (PostgreSQL) mediante el driver pgx, eliminando por completo SQLite.

--------------------------------------------------------------------------------
💡 ARQUITECTURA LÓGICA DE CAPAS
--------------------------------------------------------------------------------

1.  ENTRYPOINT (cmd/server/main.go):
    Arranca el servidor HTTP en el puerto :8080 con timeouts de seguridad, 
    inicializa el cliente persistente de Gemini y establece conexión obligatoria
    con la base de datos Supabase (PostgreSQL) a través del driver nativo pgx, 
    inyectando las dependencias necesarias. Falla catastróficamente si no se provee
    una DATABASE_URL válida en el entorno.

2.  LÓGICA DEL SISTEMA (internal/):
    Código privado que encapsula las funcionalidades principales.
    - database/: Encargada de conectarse a la nube de Supabase mediante pgx y
                 guardar de forma consistente el progreso del usuario.
    - auth/: Encargada de redirigir, intercambiar tokens y firmar cookies seguras.
    - llm/: Encargada de hablar con la API de Gemini 1.5, evaluar código de forma
            estática (prevención RCE) y generar el RAG dinámico.
    - api/: Encargada de la conexión WebSocket /ws/swarm, coordinando los payloads
            de comunicación de los mundos.

3.  PRESENTACIÓN (ui/):
    El código visual servido al navegador.
    - html/index.html: Estructura semántica de los paneles reactivos de cristal.
    - static/: Carpeta pública que expone los recursos estáticos del frontend Ciber-Zen
               (JS de control de estado, CSS de Glassmorphism e imágenes).
